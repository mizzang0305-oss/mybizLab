# MyBiz 제품 근거 정리

이 문서는 현재 저장소와 GitHub 흐름을 바탕으로 `MyBiz가 아이디어가 아니라 실제 동작형 제품`이라는 점을 보여주기 위한 근거 정리본입니다.

## 1. 제품 한 줄 정의

MyBiz는 공개 유입, 문의, 예약, 웨이팅, 주문, 설문에서 생기는 고객 신호를 고객 기억으로 묶고, AI가 이를 실행 가능한 운영 액션으로 바꾸는 소상공인 고객기억 매출 시스템입니다.

## 2. 로컬 저장소 기준 확인된 제품 범위

### 공개 유입과 전환

- 공개 랜딩/홈 진입
- `/onboarding` 기반 진단 및 스토어 생성 흐름
- 스토어별 공개 페이지
- 메뉴/주문 진입
- 공개 문의 페이지

### 운영 채널

- 문의 관리
- 예약 관리
- 웨이팅보드
- 주문 관리
- 설문 관리
- 고객 관리

### AI 및 분석

- AI 매니저
- AI 리포트
- 매출 분석

### 운영 기반

- 요금제 FREE / PRO / VIP
- 스토어 생성 요청
- 결제 및 활성화 흐름
- 공개 페이지/브랜드 설정

## 3. 데이터 구조상 확인되는 차별점

`src/shared/types/models.ts` 기준으로 MyBiz는 아래를 별도 엔터티로 관리합니다.

### 고객 기억 축

- `Customer`
- `CustomerContact`
- `CustomerPreference`
- `CustomerTimelineEvent`
- `ConversationSession`
- `ConversationMessage`
- `VisitorSession`

이 구조는 단순 고객 목록이 아니라, 고객과 매장 사이의 접점을 시간 흐름으로 누적하는 메모리 스파인에 가깝습니다.

### 입력 채널

- `Inquiry`
- `Reservation`
- `WaitingEntry`
- `Order`
- `SurveyResponse`

즉, MyBiz는 “문의만”, “예약만”, “주문만” 따로 관리하는 툴이 아니라, 서로 다른 채널을 한 고객 맥락으로 합치는 구조를 갖고 있습니다.

### 실행 및 수익화

- `AIReport`
- `BillingRecord`
- `StoreSubscription`
- `StoreRequest`
- `StoreProvisioningLog`

이 점은 단순 데모를 넘어, 실제 운영 전환과 SaaS 구조를 함께 고려했다는 근거가 됩니다.

저장소 기준 핵심 기능은 최소 12개로 확인됩니다.

## 5. MyBiz가 강한 이유

### 1) 공개 페이지에서 시작한다

많은 도구는 고객이 이미 남긴 데이터만 관리합니다.  
MyBiz는 공개 페이지에서 첫 유입을 받는 시점부터 설계되어 있습니다.

### 2) 비정형 채널을 놓치지 않는다

실제 매장에서는 문의, 예약, 웨이팅이 매우 중요하지만 보통 분절되어 남습니다.  
MyBiz는 이 채널들을 운영 중심 신호로 다룹니다.

### 3) AI가 메모리 위에 올라간다

AI를 단독 기능으로 두지 않고, 고객 기억과 운영 흐름을 기반으로 요약·분류·추천·리포트를 수행합니다.

### 4) 무료 진입 -> 유료 확장 구조가 있다

저장소의 요금제 구조상 FREE / PRO / VIP 흐름이 정리돼 있어, 시장 진입과 확장 전략을 함께 설명할 수 있습니다.

- 공개 유입 채널 퍼시스턴스

- 런타임 안정화

## 9. 공모전용 핵심 근거 문장

아래 문장은 제품 근거 설명에 바로 쓸 수 있습니다.

### 문장 1

MyBiz는 공개 스토어, 문의, 예약, 웨이팅, 주문, 설문, 고객 타임라인, AI 리포트, 결제 및 스토어 활성화까지 연결된 동작형 제품 구조를 보유하고 있습니다.

### 문장 2

저장소 기준 고객, 방문자 세션, 문의, 예약, 웨이팅, 대화 세션, 타임라인 이벤트가 별도 모델로 분리되어 있어, 고객 관계를 일회성 입력이 아니라 축적 가능한 기억 구조로 설계했음을 확인할 수 있습니다.

### 문장 3

최근 GitHub 개발 흐름에서도 고객 기억 도메인 정렬, 공개 유입 채널 퍼시스턴스, public API 안정화, 온보딩 경험 고도화, AI/운영 동반자 UX 개선이 연속적으로 반영되고 있어 제품 완성도가 빠르게 높아지고 있습니다.

## 10. 심사자료에 붙이기 좋은 스크린샷 추천

아래 화면을 붙이면 제품 스토리가 훨씬 잘 보입니다.

1. 공개 스토어 첫 화면
2. 문의 페이지
3. 예약 또는 웨이팅 화면
4. 고객 관리 화면
5. 고객 타임라인 또는 관련 고객 정보 화면
6. AI 매니저 / AI 리포트 화면
7. 온보딩 진단 흐름
8. 요금제/스토어 활성화 흐름

## 11. 현재 강점과 보완점

### 강점

- 제품 스토리가 명확함
- 고객 기억이라는 고유한 축이 존재함
- 공개 유입부터 운영 액션까지 이어짐
- 기술 구조가 데모 수준을 넘어섬
- 최근 GitHub 개선 흐름이 활발함

### 보완점

- 실제 운영 매장 사례와 수치가 아직 문서에 붙어 있지 않음
- 공모전 트랙 적합성을 위해 `대표 매장/브랜드` 중심 서사가 필요함
- “SaaS”보다 “소상공인 운영서비스”로 번역하는 작업이 필요함

## 12. 추천 증빙 문장

MyBiz는 이미 공개 유입, 고객 입력 채널, 고객 기억 축, AI 레이어, 운영 실행 화면이 이어지는 구조를 갖춘 상태이며, GitHub 기준 최근 며칠 사이에도 핵심 도메인과 공개 채널, 온보딩 경험이 지속적으로 고도화되고 있습니다. 이는 단순 아이디어가 아니라 실증 가능한 제품 단계에 들어섰다는 증거입니다.
